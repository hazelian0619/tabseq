# 注释规范（Doc/Code）

目的：让新同学在不看论文的情况下，能快速理解“设计意图 + 输入输出 + 层间关系”。  
原则：少而清晰；解释“为什么”和“怎么用”，不是重复代码字面意思。

## 1) 适用范围

- **脚本入口**（`scripts/*.py`）：必须说明用途、使用场景、输入输出。
- **数据/编码/指标核心文件**（`src/tabseq/data|labels|metrics`）：必须说明字段含义与设计原因。
- **烟测脚本**（`scripts/smoke_*.py`）：必须说明维度选择的“占位性质”与验证目标。

## 2) 模块级注释模板（文件顶部）

```text
用途：一句话说明本文件用来做什么。
设计意义：为什么需要它（解决什么问题/避免什么风险）。
使用场景：什么时候运行它。
注意：不包含/不负责的内容。
```

## 3) 变量与字段注释标准

### 3.1 维度/形状必须解释

- **说明“为什么是这个维度”**，不是只写“shape=...”
- 如果是占位值，明确写“占位，仅用于形状验证”

示例（好的）：
```text
# 3x4: 3 个样本 × 4 个数值特征，用于验证 Dataset 形状对齐（非训练规模）
```

### 3.2 字段含义必须解释

涉及下列字段时，必须有注释说明：
- `dec_input`：训练前缀，`[SOS] + y_seq[:-1]`
- `y_seq`：叶子桶索引的二进制序列
- `y_mht`：每一步的 multi-hot 叶子集合（监督目标）
- `y_raw`：原始连续 y（评估用）

## 4) 指标注释标准

涉及指标时，必须说明：
- **意义**：这个指标“衡量什么”
- **趋势**：数值越大/越小代表什么
- **边界**：只用于诊断还是用于最终比较

示例（好的）：
```text
# PICP：区间覆盖率，期望接近 confidence（越高越可靠，但可能更保守）
# MPIW：区间平均宽度，越小越尖锐（但覆盖率可能下降）
```

## 5) 层间关系注释（强制）

当文件连接多个模块时，必须说明“上一层→这一层→下一层”的关系。

示例（好的）：
```text
# TraceLabelEncoder 产出 y_seq/y_mht
# TabSeqDataset 用 y_seq 构造 dec_input，并把 y_mht 作为监督目标
# 模型训练时用 dec_input 预测 y_mht
```

## 6) 禁止事项

- 只写“shape=...”不解释为什么
- 重复代码字面含义（如“assigns value to x”）
- 用英文缩写不解释（首次出现需中文解释）

## 7) 记忆法（可选）

- “y 先变序列，序列喂模型；模型出分布，分布算指标”
- “训练看 y_mht，评估看 y_raw；dec_input 是训练前缀”
